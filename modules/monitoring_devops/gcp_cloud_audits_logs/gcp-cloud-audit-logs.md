# Google Cloud Audit Logs

[Cloud Audit Logs](https://cloud.google.com/logging/docs/audit) provide a record of **who did what, where, and when** across your Google Cloud resources. They are part of Cloud Logging and are essential for security investigations, compliance auditing, and change tracking. Audit logs are automatically generated by GCP services and cannot be disabled for Admin Activity events.

> Back to [GCP Module & Service Hierarchy](../../../gcp-module-service-list.md)

---

## Overview

| Capability | Description |
|------------|-------------|
| **Admin Activity** | Who created, modified, or deleted resources — always on, no charge |
| **Data Access** | Who read or wrote resource data — off by default, configurable |
| **System Events** | Automated GCP actions (live migration, auto-scaling) — always on, no charge |
| **Policy Denied** | Requests blocked by org policy or VPC Service Controls — always on, no charge |
| **Immutable `_Required` bucket** | Admin Activity and System Event logs stored for 400 days, cannot be deleted |
| **Org-level aggregation** | Export all project audit logs via org/folder sinks |

## Core Concepts

### Audit Log Types

| Type | `logName` | Default | Charged |
|------|-----------|---------|---------|
| **Admin Activity** | `cloudaudit.googleapis.com/activity` | Always enabled | No |
| **Data Access** | `cloudaudit.googleapis.com/data_access` | Disabled | Yes |
| **System Event** | `cloudaudit.googleapis.com/system_event` | Always enabled | No |
| **Policy Denied** | `cloudaudit.googleapis.com/policy` | Always enabled | No |

### Enabling Data Access Logs

Data Access logs must be explicitly enabled per service and access type. Enable at project, folder, or org level:

```hcl
# Enable all Data Access log types for Cloud Storage at project level
resource "google_project_iam_audit_config" "gcs_data_access" {
  project = var.project_id
  service = "storage.googleapis.com"

  audit_log_config {
    log_type = "DATA_READ"
  }
  audit_log_config {
    log_type = "DATA_WRITE"
  }
}

# Enable for all services at org level
resource "google_organization_iam_audit_config" "all_data_access" {
  org_id  = var.org_id
  service = "allServices"

  audit_log_config {
    log_type = "ADMIN_READ"
  }
  audit_log_config {
    log_type = "DATA_READ"
    exempted_members = [
      "serviceAccount:trusted-sa@my-project.iam.gserviceaccount.com"
    ]
  }
}
```

### Audit Log Entry Structure

Each audit log entry contains a `protoPayload` of type `AuditLog`:

| Field | Description |
|-------|-------------|
| `protoPayload.methodName` | API method called (e.g., `storage.objects.get`) |
| `protoPayload.resourceName` | Full resource path |
| `protoPayload.authenticationInfo.principalEmail` | Caller identity |
| `protoPayload.requestMetadata.callerIp` | Source IP address |
| `protoPayload.status` | gRPC status code (0 = success) |
| `protoPayload.serviceName` | The GCP service (e.g., `storage.googleapis.com`) |
| `resource.type` | Resource type (e.g., `gcs_bucket`) |
| `timestamp` | When the operation occurred |

### Querying Audit Logs

Using Log Analytics (SQL) or the Logs Explorer filter syntax:

```sql
-- Find all IAM policy changes in the last 7 days
SELECT
  timestamp,
  proto_payload.audit_log.authentication_info.principal_email AS actor,
  proto_payload.audit_log.method_name AS action,
  proto_payload.audit_log.resource_name AS resource
FROM `my-project.global._Default._AllLogs`
WHERE
  log_id = "cloudaudit.googleapis.com/activity"
  AND proto_payload.audit_log.method_name LIKE "%SetIamPolicy%"
  AND timestamp > TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 7 DAY)
ORDER BY timestamp DESC
```

Logs Explorer filter:

```
logName="projects/my-project/logs/cloudaudit.googleapis.com%2Factivity"
protoPayload.methodName="SetIamPolicy"
```

### Exporting Audit Logs

Route audit logs to a centralized security project:

```hcl
# Org-level sink — captures audit logs from ALL projects
resource "google_logging_organization_sink" "audit_to_bq" {
  name             = "org-audit-logs"
  org_id           = var.org_id
  destination      = "bigquery.googleapis.com/projects/${var.security_project}/datasets/audit_logs"
  filter           = "logName:(\"cloudaudit.googleapis.com/activity\" OR \"cloudaudit.googleapis.com/data_access\")"
  include_children = true

  bigquery_options {
    use_partitioned_tables = true
  }
}

resource "google_bigquery_dataset_iam_member" "sink_writer" {
  project    = var.security_project
  dataset_id = "audit_logs"
  role       = "roles/bigquery.dataEditor"
  member     = google_logging_organization_sink.audit_to_bq.writer_identity
}
```

## Exemptions

Exclude high-volume trusted service accounts from Data Access logging to reduce ingestion cost:

```hcl
resource "google_project_iam_audit_config" "bq_data_access" {
  project = var.project_id
  service = "bigquery.googleapis.com"

  audit_log_config {
    log_type = "DATA_READ"
    exempted_members = [
      "serviceAccount:etl-pipeline@my-project.iam.gserviceaccount.com"
    ]
  }
}
```

## Security Guidance

- Enable **Data Access logs for Secret Manager** and **Cloud KMS** to capture every secret read and encryption operation — these are critical for security forensics.
- Use an **org-level sink with `include_children = true`** to ensure no project can escape centralized audit log collection.
- Route logs to a **separate security project** where application developers have no write/delete access.
- Use **partitioned BigQuery tables** in the sink destination (`use_partitioned_tables = true`) to control query cost.
- Alert on `SetIamPolicy`, `CreateServiceAccountKey`, and `CreateServiceAccount` via log-based metrics + Cloud Monitoring.
- Exempt only well-understood, high-volume service accounts from Data Access logging — document every exemption.
- Never disable `cloudresourcemanager.googleapis.com` Admin Activity logs — they capture all hierarchy changes.
